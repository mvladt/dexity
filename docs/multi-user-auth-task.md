# Мульти-пользовательская авторизация — Task

## Контекст

Сейчас в Dexity один пользователь (жёстко `id=1` в БД) и единственная защита — общий `ACCESS_TOKEN` в `.env` (shared secret на всё приложение). LLM-вызовы идут через личный `YC_API_KEY` пользователя без каких-либо лимитов на потребление.

Возник при обсуждении деплоя на прод (`docs/vps-deploy-cicd-plan.md`, Этап 2): решили не блокировать личный прод-деплой ради этой фичи, но зафиксировать требования отдельно, так как в ближайшем будущем хочется открыть Dexity другим людям.

Два независимых трека:
1. **Деплой для себя** — текущая модель (`ACCESS_TOKEN`) не меняется, идёт своим ходом по плану деплоя.
2. **Эта таска** — предусловие для *публичного* открытия доступа другим людям. Не блокирует пункт 1.

## Зафиксированные решения

- [ ] **Регистрация** — invite-only. Владелец (я) генерирует одноразовую ссылку-токен, отправляет человеку лично. Публичной самостоятельной регистрации нет.
- [ ] **Вход** — OAuth через Yandex ID. Без пароля, без Google/др. провайдеров. Invite-токен при первом переходе по ссылке привязывается к Yandex ID пользователя.
- [ ] **Оплата LLM-вызовов** — общий пул: один `YC_API_KEY` (мой) на всех пользователей. Свои ключи пользователи не вводят.
- [ ] **Лимиты** — токены (prompt + completion) в день/месяц на пользователя. Считаем от существующих `messages.prompt_tokens` / `completion_tokens`.
- [ ] **Администрирование** — без UI. Управление пользователями/инвайтами/лимитами вручную через БД или CLI-скрипт.
- [ ] **Данные** — при выкатке новой схемы текущая БД сбрасывается (личные чаты не мигрируются, начинаем с чистой).

## Открытые вопросы для проектирования

1. **OAuth-флоу Yandex ID** — конкретный протокол (Yandex OAuth 2.0 / OpenID Connect), нужный scope (`login:email` / `login:info`), что делаем с access/refresh токеном Yandex после идентификации (нужен ли он дальше или только для получения identity при первом входе).
2. **Invite-токен** — формат (случайная строка, длина/энтропия), TTL, где хранится (новая таблица `invites`), как инвалидируется после использования (одноразовость), что если ссылка просрочена.
3. **Сессия после логина** — чем заменяется `Authorization: Bearer <ACCESS_TOKEN>`. Варианты: server-side сессия + httpOnly cookie, или JWT. Нужно определиться с TTL сессии и способом логаута.
4. **Схема БД** — расширение `users` (yandex_id, email/login, created_at), новая `invites` (token, created_by, used_by, expires_at, used_at), новая `usage`/агрегация лимитов (период, потрачено токенов) — считать на лету по `messages` или вести отдельный счётчик.
5. **Реализация лимита** — где проверяется (перед вызовом LLM), что происходит при превышении (429 + сообщение в UI), сброс по календарным суткам/месяцу или скользящее окно.
6. **Миграция** — снести текущие таблицы и накатить новую схему через `migrate.ts` (без скрипта переноса данных, см. решение выше).
