# Промт: спецификация AI-чата (клон Perplexity.ai)

Я хочу создать минимальный, но полностью функциональный AI-чат для личного использования — наивный клон Perplexity.ai по UX/дизайну.

---

## Стек

**Frontend:** React 18, TypeScript, Vite, Zustand

- UI/дизайн-система: `@gravity-ui/uikit` + `@gravity-ui/aikit` (https://gravity-ui.com/ru/libraries/aikit)
- Все UI-компоненты берём из GravityUI/AIKit. Кастомный компонент создаётся только если в этих библиотеках нет подходящего аналога — это должно быть явно обосновано в спецификации.

**Backend:** Node.js, TypeScript, Fastify, SQLite (Drizzle ORM), Zod для валидации

**LLM:** Yandex Cloud AI Studio, OpenAI-совместимый API:

- `base_url = "https://ai.api.cloud.yandex.net/v1"`
- Клиент: `openai` npm SDK
- Дефолтная модель: `qwen3.6-35b-a3b`
- Полный model id: `gpt://<folderId>/qwen3.6-35b-a3b/latest` — `folderId` подставляется из `YC_FOLDER_ID` в `.env`
- Переключение модели через `MODEL_ID` в `.env`

**Деплой:** VPS Ubuntu (Timeweb), домен `dexity.mvladt.ru`, Nginx reverse proxy

## Аутентификация

**Прототип (реализовать сейчас):** один `ACCESS_TOKEN` в `.env` на бэке. Фронт отправляет его в заголовке `Authorization: Bearer <token>`. Бэк проверяет через Fastify middleware.

**Будущее расширение (не реализовывать, только заложить в схему):** каждый пользователь вводит собственные `folderId` и `apiKey` от Yandex Cloud. Учесть в схеме БД и TypeScript-интерфейсах.

## Функционал MVP

- Логин по токену (guard на фронте + middleware на бэке)
- Главная страница в стиле Perplexity: большой центрированный инпут, подсказки/саджесты
- Сайдбар с историей чатов — создание, переименование, удаление
- Стриминг ответа через SSE (побуквенный вывод)
- **Контекст хранится самостоятельно:** бэк загружает полную историю сообщений из SQLite, формирует `messages[]` и передаёт в LLM API при каждом запросе
- История сообщений в SQLite — и для контекста LLM, и для отображения в UI
- Рендеринг Markdown + подсветка кода (через `MarkdownRenderer` из AIKit)
- Системная тема (`prefers-color-scheme`) + ручной переключатель через `ThemeProvider` из `@gravity-ui/uikit`

## Маппинг AIKit-компонентов (обязательно включить в спецификацию)

Для каждого экрана/блока явно указать, какой компонент AIKit/GravityUI используется:

| Блок                     | Компонент                              |
| ------------------------ | -------------------------------------- |
| Весь чат-лейаут          | `ChatContainer`                        |
| Список сообщений         | `MessageList`                          |
| Сообщение ассистента     | `AssistantMessage`                     |
| Сообщение пользователя   | `UserMessage`                          |
| Поле ввода               | `PromptInput`                          |
| История чатов в сайдбаре | `History`                              |
| Пустое состояние         | `EmptyContainer`                       |
| Стриминг / ожидание      | `ThinkingMessage`, `Shimmer`           |
| Рендеринг Markdown       | `MarkdownRenderer`                     |
| Тема                     | `ThemeProvider` из `@gravity-ui/uikit` |

Если какой-то блок не покрывается AIKit — зафиксировать явно и предложить решение.

## Подход: SDD + AI-First

**SDD:** сначала полная спецификация, потом код.

**AI-First:** спецификация пишется для AI-агента (Claude, Cursor Agent). Каждый модуль описан так детально, чтобы агент писал код файл за файлом без архитектурных решений на своей стороне. Никаких "выберите подходящий подход" — только конкретные решения.

## Содержание спецификации

1. **Структура проекта** — monorepo (`/client`, `/server`), структура папок (FSD для фронта, feature-modules для бэка)
2. **Схема БД** — все таблицы, поля, типы, индексы, связи. С учётом будущего мульти-пользователя
3. **API-эндпоинты** — метод, путь, тело запроса/ответа (TypeScript-типы), коды ошибок
4. **TypeScript-интерфейсы** — общие типы: сообщения, чаты, пользователь, SSE-события, типы AIKit
5. **ENV-переменные** — все переменные фронта и бэка с описанием и примерами значений
6. **Компоненты фронта** — список с пропсами, стейтом, поведением. Для каждого — какой AIKit/GravityUI компонент используется как основа
7. **Zustand-сторы** — структура, экшены, взаимодействие с API
8. **Поток стриминга** — детально: бэк проксирует SSE от Yandex → фронт читает поток → UI обновляется. Как именно AIKit `AssistantMessage`/`ThinkingMessage` подключаются к стриму
9. **Поток контекста** — бэк загружает историю из БД, формирует `messages[]`, подставляет `folderId` в model id, отправляет в LLM
10. **Nginx-конфигурация** — проксирование `/api` на бэк, `/` на фронт, поддержка SSE (`proxy_buffering off`)

После спецификации жди моего подтверждения. Только потом реализуй код — пошагово, файл за файлом, начиная с бэка.
