# Задача — `web_fetch`: распознавание страниц-капчи

## Контекст

Найдено при разборе чата 229 (запрос «свежие новости о Яндексе»). Модель вызывала
`web_fetch` на `yandex.ru/...`, сервис редиректил на страницу капчи
`https://yandex.ru/showcaptcha?...`, а `@mozilla/readability` извлекал её как
**валидный «контент»** (`error:false`, текст вроде «Подтвердите, что запросы
отправляли вы, а не робот»). Модель получала мусор, считала источник
«заблокированным» и **зацикливалась** на повторных `web_search` / `web_fetch`,
выжигая раунды (`MAX_TOOL_ROUNDS`).

Симптом для пользователя — тупиковый чат без финального ответа. Сам тупик уже
подстрахован фолбэком в `server/src/routes/messages.ts` (коммит `37c5eaa`), но
**первопричина — здесь**: `web_fetch` не отличает капчу от настоящей страницы.

## Цель

`fetchUrl()` должен распознавать страницы-капчи (и редиректы на них) и возвращать
модели **явную ошибку** вместо извлечённого мусора, чтобы модель не ретраила тот
же URL и быстрее переходила к доступным источникам.

## Где править

- `server/src/services/fetch.ts` → `fetchUrl()`:
  - финальный URL после редиректов лежит в `currentUrl` (строки ~190–221);
  - извлечённый текст — в `article.textContent` / `content` (строки ~257–268).

## Сигналы капчи (эвристики)

- [ ] Финальный URL содержит `showcaptcha` / `/captcha` / `smartcaptcha` (хост Яндекса
      редиректит именно на `/showcaptcha`). Это самый надёжный признак — проверять
      сразу после цикла редиректов, до парсинга.
- [ ] Короткий/типовой текст-маркер в извлечённом контенте: «подтвердите, что вы не
      робот», «are you a robot», «captcha», «нам очень жаль» и т.п. (регистронезависимо).
      Запасной сигнал на случай, когда капча отдаётся без отдельного URL.

## Реализация

- [ ] Добавить хелпер `isCaptchaUrl(url: string): boolean` (проверка по подстрокам пути/хоста).
- [ ] После цикла редиректов: если `isCaptchaUrl(currentUrl)` — бросать
      понятную ошибку, например `Страница защищена капчей: ${hostname}`.
- [ ] (Опционально) После извлечения контента — проверка текста по списку маркеров;
      если совпало и контент короткий (< N символов) — та же ошибка.
- [ ] Сообщение ошибки писать так, чтобы модели было ясно: **не ретраить этот URL**,
      а взять другой источник.

## Критерии приёмки

- [ ] `web_fetch` на URL, редиректящий на `yandex.ru/showcaptcha?...`, возвращает
      `error` (а не «успешный» мусорный контент).
- [ ] В сценарии «новости о Яндексе» модель не зацикливается на капче и формирует
      ответ за разумное число раундов (либо честно сообщает, что источники недоступны).
- [ ] Обычные страницы (без капчи) по-прежнему читаются корректно — нет ложных
      срабатываний на словах «captcha»/«робот» в нормальных статьях (отсюда условие
      «короткий контент» для текстового сигнала).
- [ ] `npx tsc --noEmit` в `server/` чистый.

## Вне скоупа
