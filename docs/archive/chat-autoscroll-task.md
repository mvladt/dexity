# Автоскролл к новым сообщениям в чате

**Дата:** 2026-05-21
**Закрыта:** 2026-05-26
**Компонент:** `client/src/components/ChatStream.tsx`, `client/src/styles.css` (`.chat-messages`)

## Резолюция

Scope сокращён до **sticky-bottom**, без anchor-to-top user-сообщения и без FAB-кнопки «↓ К новым сообщениям». Если поведение «как в Perplexity» (свежее сообщение к верхнему краю + spacer 70vh) понадобится — оформим отдельной задачей.

Финал:

- `f37ac24` — первая самописная версия sticky-scroll.
- `9eeb936` — переписана отвязка через сравнение `scrollTop` с программно установленным значением.
- `112fed5` — самописный код заменён на `useSmartScroll` из `@gravity-ui/aikit` (минус ~30 строк, MutationObserver вместо хака с `partialContent`-deps).

## Проблема

Когда контент `.chat-messages` переполняет контейнер, появляется вертикальный скроллбар (это ожидаемо). Но при появлении нового сообщения и во время стриминга контейнер **не подскролливается** — новые сообщения уезжают за нижний край и становятся видны только после ручного скролла. Неудобно: пользователь не видит ни своё только что отправленное сообщение, ни ответ ассистента.

## Ожидаемое поведение (как в Perplexity / ChatGPT)

Две связки поведения:

1. **При отправке нового user-сообщения** — скролл так, чтобы свежее user-сообщение оказалось у **верхнего края** видимой области, а ответ ассистента разворачивался ниже. Для этого под последним сообщением держится **spacer** ~`70vh`, иначе в конце беседы скролла «не хватит» чтобы поднять сообщение вверх.
2. **Во время стриминга** — **sticky bottom**: если пользователь у нижнего края (`scrollHeight - scrollTop - clientHeight < ~80px`), контейнер автоматически скроллит вниз по мере прихода токенов. Если пользователь ушёл вверх читать — автоскролл **выключается** до явного действия (не вырываем фокус). Опционально — плавающая кнопка «↓ К новым сообщениям», когда `!isAtBottom && streaming`.

Главное: отслеживается **намерение пользователя** через флаг `isAtBottom`.

## План

- [ ] В `ChatStream` повесить `ref` на скролл-контейнер `.chat-messages` (обернуть `MessageList`, либо использовать ref на внешний div)
- [ ] Локальный стейт `isAtBottom: boolean`, обновляется в `onScroll` контейнера (порог ~80px от низа)
- [ ] `useLayoutEffect` по `messages.length`:
  - запомнить id последнего user-сообщения, через `ref`/`querySelector` найти его DOM-узел и `scrollIntoView({ block: 'start', behavior: 'smooth' })`
- [ ] `useLayoutEffect` по `partialContent`:
  - если `isAtBottom` — `el.scrollTop = el.scrollHeight`
  - иначе ничего не делаем
- [ ] Добавить spacer под списком сообщений (`min-height: 70vh` у последнего блока **или** пустой `<div>` с `flex: 0 0 70vh` в конце ленты)
- [ ] (опционально) Кнопка «↓ К новым сообщениям» в правом нижнем углу `.chat-messages` при `!isAtBottom && streaming`
- [ ] Проверить мобильный вьюпорт: `70vh` под последним сообщением не должен мешать чтению коротких бесед

## Acceptance Criteria

- [ ] После отправки сообщения пользователь видит **своё сообщение у верхнего края** и начало стриминга ответа ниже
- [ ] Во время стриминга при позиции «у дна» — контейнер сам подскролливает, новый текст всегда виден
- [ ] Если пользователь во время стриминга проскроллил вверх — автоскролл **не происходит**, фокус остаётся на том, что читает пользователь
- [ ] Загрузка существующего чата — открывается у дна (последнее сообщение видно)
- [ ] Нет повторного срабатывания `window`-скролла из `bug-report-scroll-bottom.md` (скроллится только `.chat-messages`, не `document`)

## Открытые вопросы

- `MessageList` из `@gravity-ui/aikit` — у него внутри свой скролл-контейнер или скроллится наш `.chat-messages`? Проверить перед реализацией, от этого зависит, куда вешать `ref` и слушатели.
- `behavior: 'smooth'` или `'auto'` для anchor-to-top? Perplexity делает smooth, но это может мешать во время быстрых ответов модели.
