# Задача — `web_fetch`: распознавание страниц-капчи

## Контекст

Найдено при разборе чата 229 (запрос «свежие новости о Яндексе»). Модель вызывала
`web_fetch` на `yandex.ru/...`, сервис редиректил на страницу капчи
`https://yandex.ru/showcaptcha?...`, а `@mozilla/readability` извлекал её как
**валидный «контент»** (`error:false`, текст вроде «Подтвердите, что запросы
отправляли вы, а не робот»). Модель получала мусор, считала источник
«заблокированным» и **зацикливалась** на повторных `web_search` / `web_fetch`,
выжигая раунды (`MAX_TOOL_ROUNDS`).

Симптом для пользователя — тупиковый чат без финального ответа. Сам тупик уже
подстрахован фолбэком в `server/src/routes/messages.ts` (коммит `37c5eaa`), но
**первопричина — здесь**: `web_fetch` не отличает капчу от настоящей страницы.

## Цель

`fetchUrl()` должен распознавать страницы-капчи (и редиректы на них) и возвращать
модели **явную ошибку** вместо извлечённого мусора, чтобы модель не ретраила тот
же URL и быстрее переходила к доступным источникам.

## Где править

- `server/src/services/fetch.ts` → `fetchUrl()`:
  - финальный URL после редиректов лежит в `currentUrl` (строки ~190–221);
  - извлечённый текст — в `article.textContent` / `content` (строки ~257–268).

## Сигналы (итог)

Изначально планировался текстовый маркер-список («не робот», «captcha» и т.п.) как
запасной сигнал. **Отказались**: список конечный и хрупкий (языки, формулировки),
ловит только частный случай. Вместо него — универсальный порог длины: fetch
существует, чтобы дать модели текст; дал мало текста — бесполезен, неважно почему
(капча, пустая SPA, paywall-превью, заглушка). Один порог закрывает весь класс.

- [x] Финальный URL содержит `showcaptcha` / `/captcha` / `smartcaptcha` — проверка
      сразу после цикла редиректов, до парсинга. Даёт точную диагностику «капча»
      в плашке fetch и экономит работу Readability.
- [x] Порог длины извлечённого контента (`CONTENT_MIN = 200`) — основной
      универсальный сигнал. Заменил текстовые маркеры; ловит и капчу без URL-признака.

## Реализация

- [x] Добавлен хелпер `isCaptchaUrl(url: string): boolean` (подстроки `showcaptcha` /
      `/captcha` / `smartcaptcha`).
- [x] После цикла редиректов: `isCaptchaUrl(currentUrl)` → ошибка с hostname.
- [x] После извлечения: `content.length < CONTENT_MIN` → ошибка с числом символов.
- [x] Тексты ошибок явно говорят модели: **не повторять запрос к URL**, взять другой источник.

## Критерии приёмки

- [x] `web_fetch` на URL, редиректящий на `yandex.ru/showcaptcha?...`, возвращает
      `error` (а не «успешный» мусорный контент).
- [ ] В сценарии «новости о Яндексе» модель не зацикливается на капче и формирует
      ответ за разумное число раундов (проверить вручную в UI на живом запросе).
- [x] Обычные страницы (без капчи) по-прежнему читаются — ложных срабатываний нет:
      основной сигнал по URL, а порог длины (200) ниже объёма любой реальной статьи.
- [x] `npx tsc --noEmit` в `server/` чистый.
