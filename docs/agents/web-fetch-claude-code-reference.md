# Как устроен `web_fetch` в Claude Code — что брать к нам

Референс: как реализован WebFetch в Claude Code (по разбору mikhail.io, giuseppegurgone, доке Anthropic). Сравнение с нашим планом `docs/web-fetch-task.md` и рекомендации.

## Как это работает у Claude (кратко)

`web_fetch` принимает **`url` + `prompt`** и возвращает **ответ на prompt**, а не сырой текст страницы:

```
url + prompt
  → [кэш 15 мин]
  → загрузить статический HTML
  → HTML → Turndown → Markdown
  → [доверенный сайт + text/markdown + <100k символов?] → fast path (минуя модель)
  → [иначе] → малая модель (Haiku) извлекает инфу
  → ответ в основную модель
```

Ключевое:
- Конвертация — **Turndown** (HTML → Markdown), не Firefox Readability.
- Для документации — fast path; остальное переваривает малая модель.
- Запрос идёт на **remote-инстанс** (отдельная модель) — защита от prompt injection.
- JS-рендеринг **не поддерживается**, только статический HTML.

## Чем это отличается от нашего плана

Наш план: Readability → `article.textContent` (plain text, 15k символов) → дамп **прямо в основную модель**.
У Claude: страница переваривается малой моделью, в основную идёт уже **digest**.

## Рекомендации

### 1. Readability + Turndown → markdown (брать в v1)

Мы клон Perplexity и постоянно читаем **документацию**, где главное — код-блоки, таблицы, заголовки. Наш план «только plain text» (`article.textContent`) их сплющивает: код теряет форматирование, таблицы — в кашу.

Чистый Turndown на всём HTML затащит nav/footer. Сильная комбинация:

**Readability (изолировать тело статьи `article.content`) → Turndown (этот фрагмент → Markdown).**

Одна маленькая зависимость, на выходе чистый структурированный markdown. **Противоречит пункту плана «только plain text в v1» — предлагаю поменять.**

Оговорка: markdown с разметкой дороже plain-text по токенам на тот же объём. Для контента, где структура и есть смысл (дока, таблицы), оправдано. Для новостной статьи разница копеечная. Стоит померить.

### 2. `url + prompt` + малая модель-прокладка (в v2)

У Claude малая модель (Haiku) переваривает страницу до того, как контент попадёт в основную модель. Зачем нам:
- **Экономия контекста.** 15k символов × параллельные фетчи × несколько раундов round-loop'а — основной контекст распухает мгновенно. Pre-digest держит его тощим.

Для MVP — смена архитектуры (+1 LLM-вызов на fetch, +латентность, зависимость от дешёвой модели; у Yandex есть YandexGPT Lite). **Откладываем в v2, фиксируем как оптимизацию контекста.** Знать про потолок по контексту нужно заранее.

### 3. Prompt injection — в `TODO.md` (брать сейчас)

В нашем плане вектор не закрыт никак: льём сырой текст чужой страницы прямо в контекст основной модели → страница с «ignore previous instructions» может угнать модель.

Remote-инстанс (как у Claude) — оверкилл, мы не платформа, юзер один. Но сам факт, что **`web_fetch` — это канал prompt injection в основную модель**, фиксируем в `TODO.md` рядом с DNS rebinding. Схема из п.2 (малая модель-прокладка) попутно closes этот вектор — ещё один довод за digest в v2.

## Что НЕ тащить

- **Remote-инстанс / отдельный сервис** — не наш масштаб.
- **Кэш 15 минут между ответами** — наш «кэш в рамках одного ответа» для личного использования достаточен.
- JS-рендеринг Claude тоже не делает (статический HTML only) — **подтверждает валидность нашего скоупа**.
