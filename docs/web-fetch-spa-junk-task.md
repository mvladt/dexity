# web_fetch: SPA отдаёт JSON-стейт вместо текста — задача

**Статус:** отложено (редкий кейс, ~раз в полгода). Анализ зафиксирован, код не пишем.

## Проблема

`web_fetch` на SPA без SSR (пример — Яндекс Маркет) возвращает **успех** с
большим объёмом текста, но этот «текст» — внутренний JSON-стейт приложения из
inline-`<script>`, а не контент страницы. Readability не находит статью и
выдирает сериализованный стейт как «основной текст».

Обнаружено в чате 252: два fetch на `market.yandex.ru/search?...` помечены
зелёными «11 747 симв.» / «11 734 симв.», но внутри:

```
{"widgets":{"@marketfront/Fink":{"/content/page/fancyPage/fink":
{"event":"mount","zoneData":{"expFlags":{"all_apphost-cpr-tabs":true, ...
```

Модель сама поняла, что данных нет («страницы содержат динамический контент»), и
дала общий ответ — но 23 КБ мусора ушло в контекст под видом успеха.

## Почему текущие защиты не ловят

- `CONTENT_MIN = 200` — мимо: мусора **много** (11–13 КБ).
- `isCaptchaUrl` — мимо: это не капча, URL обычный `/search`.

Это третий класс проблемы: не капча и не пустая страница, а **много машинного
мусора**.

## Почему НЕ headless-рендер (Playwright/Chromium)

Рассматривали «дорендерить SPA и потом в Readability». Отклонено:

- **Вес:** Chromium ~300 МБ — смена класса сервиса, нарушение минимализма.
- **Латентность:** холодный старт + рендер = 3–8 с/страница, убивает агентный цикл.
- **SSRF:** headless сам резолвит DNS и ходит за сабресурсами мимо нашей ручной защиты.
- **Не решает:** Маркет детектит headless (антибот → капча), а Readability на
  каталоге товаров (grid, не статья) всё равно даст мусор.

Headless — отдельный большой трек (см. TODO «Pro Search», #23), не правка в лёгкий fetch.

## Предлагаемое решение (когда дойдут руки)

Эвристика «проза vs машинный дамп» по **доле структурных символов** `{ } [ ] " : ,`.
Ключевая мысль: пробелы форматирование меняет, а структурные символы JSON
неустранимы.

Замеры на реальных данных (мусор из чата 252 + наши `.md` как проза):

| образец                     | буквы % | пробелы % | структур. % | `":` пар/1k |
|-----------------------------|--------:|----------:|------------:|------------:|
| JSON-мусор (Маркет)         |    49.2 |       0.0 |        23.8 |        49.9 |
| проза: overview.md          |    45.0 |      34.3 |         1.3 |         0.3 |
| проза: backend.md           |    58.7 |      20.9 |         4.1 |         3.1 |
| проза: CLAUDE.md            |    71.1 |      14.1 |         2.0 |         0.0 |
| минифиц. package-lock       |    46.0 |      27.4 |        13.6 |        22.9 |

Выводы:
- **Пробелы и буквы не разделяют** (мусор Маркета: 0% пробелов, но 49% букв).
- **Структурные символы разделяют чисто:** проза ≤ 4.1%, машинное ≥ 13.6%.
- Readability отдаёт чистый `textContent` без markdown-разметки → реальная проза
  ещё ниже наших `.md`-сэмплов. В бою запас только больше.

Реализация: константа порога (≈10%) + проверка после нормализации контента в
`server/src/services/fetch.ts`, рядом с `CONTENT_MIN`. При превышении — бросать
ошибку «страница не содержит читаемого текста (динамический контент)», как для
капчи. Порог консервативный: лучше пропустить пограничный мусор, чем зарубить
настоящую статью.
