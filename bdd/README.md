# BDD — поведенческие сценарии Dexity

Общая папка для тест-сценариев по любой области приложения: аутентификация, история чатов, настройки, стриминг, UI, web search, поведение LLM и т.д. Сценарии описаны на Gherkin'е и должны быть **достаточно конкретны**, чтобы по ним мог работать:

- **человек** — как чек-лист при ручном прогоне;
- **агент-тестер** — браузерным автоматом (`playwright-cli`);
- **разработчик/агент** — как ТЗ для написания `playwright`-тестов.

## Принципы написания сценариев

- **Конкретика** — точный текст запроса/кнопки/маркера, а не «отправь что-нибудь похожее».
- **Проверяемые `Тогда`** — «в DOM есть текст X», «в БД лежит N записей», «SSE-лог содержит событие Y». Размытые формулировки («ответ корректен») — только под `@llm-judge`.
- **Один сценарий — одна проверка**. Не нагромождай.
- **`Предыстория`** — для общих `Дано`, повторяющихся во всех сценариях файла.

## Теги

Теги (`@что-то` над `Функция:` или `Сценарий:`) — метаданные для фильтрации и понимания типа проверки. В Dexity договариваемся о таких:

**Тип проверки:**

- `@e2e` — детерминированный ассёршен, годится для playwright-теста.
- `@llm-judge` — субъективная оценка ответа модели, нужен LLM-судья или человек.
- `@manual` — проверяется только глазами (вёрстка, анимации, ощущения).

**Область:**

- `@auth`, `@chat`, `@history`, `@settings`, `@streaming`, `@web-search`, `@ui`, `@persistence`, `@llm` — выбирай существующий или заводи новый, поддерживай актуальный список ниже.

**Свойства:**

- `@model-sensitive` — результат зависит от выбранной модели (актуально для `@llm-judge`).
- `@slow` — сценарий выполняется долго.
- `@wip` — черновик, временно пропустить.

## Структура

```
bdd/
├── README.md                       — этот файл
├── features/
│   ├── llm-web-search-auto.feature       — пример: автовызов web_search
│   ├── auth-login.feature                 (TODO)
│   ├── chat-history.feature               (TODO)
│   └── …
└── results/
    └── _template.md                — шаблон отчёта о прогоне
```

Имя feature-файла = `<область>-<что-проверяется>.feature`. Плоско, без вложенных папок — минимализм.

## Как прогонять

Раннера пока нет — feature-файлы это **спецификация**. Варианты прогона:

1. **Руками** — открываешь Dexity, идёшь по сценарию как по чек-листу, отмечаешь pass/fail в `results/`.
2. **Через агента-тестера** — кидаешь агенту feature-файл и просишь прогнать через `playwright-cli`, результат записать в `results/`.
3. **Сгенерировать playwright-тесты** — feature как ТЗ, агент пишет `.spec.ts` под `@e2e`-сценарии, `@llm-judge` оставляет для ручного/LLM-судьи.

Если со временем сценариев накопится много — добавим `@cucumber/cucumber` + playwright-runner или `promptfoo` для LLM-evals. Сейчас минималистично.
