# BDD — поведенческие требования к LLM в Dexity

Эти `.feature`-файлы описывают, **как должна вести себя модель** в контексте Dexity. Используются как чек-лист, когда:

- меняешь дефолтную модель;
- решаешь, добавлять ли модель в список `client/src/models.ts`;
- хочешь оценить, насколько новая модель «подходит» проекту.

Это **не e2e-тесты кода** — багу в backend/frontend feature-файл не поймает. Это спецификация поведения **модели**.

## Что внутри

- `features/web-search-auto.feature` — модель сама зовёт `web_search`, когда нужно.
- `features/web-search-suppression.feature` — модель НЕ зовёт `web_search`, когда не нужно.
- `features/multi-round-search.feature` — модель умеет дробить тяжёлый вопрос на несколько поисков.
- `features/formatting.feature` — модель форматирует ответ markdown'ом.
- `features/context-awareness.feature` — модель учитывает историю диалога.
- `features/refusal-honesty.feature` — без `web_search` модель честно говорит, что не знает свежего.

## Как прогонять

Раннера нет — Gherkin-файлы тут как **читаемая спецификация**. Прогон:

1. Открываешь Dexity, выбираешь модель.
2. Идёшь по сценариям из feature-файла как по чек-листу.
3. Записываешь результат в `results/<model>-<YYYY-MM-DD>.md` (шаблон в `results/_template.md`).

Альтернатива — попросить агента (меня) прогнать сценарии через `playwright-cli` и заполнить result.

Если позже захочется автоматизации — `promptfoo` с прямыми API-вызовами к серверу подойдёт лучше, чем cucumber+browser.
